SECCIÓN 1: INTRODUCCIÓN (ULTRA EXPANDIDA)

Descripción Completa del Sistema

El presente sistema es una plataforma web reactiva y asíncrona concebida para la gestión, auditoría y control del patrimonio financiero personal. La arquitectura del software abstrae la complejidad de la contabilidad moderna mediante interfaces interactivas, cálculos automatizados y recolección de micro-datos transaccionales. El sistema no solo actúa como un libro mayor, sino como una herramienta analítica capaz de emitir recomendaciones dinámicas (como el método 50/30/20) y consolidar el panorama económico global del usuario.

Contexto del Problema Financiero en Usuarios

A nivel global, la "ceguera financiera" —la incapacidad de rastrear el flujo de caja de microgastos y obligaciones recurrentes— resulta en endeudamiento sistémico y pobre historial crediticio. La mala gestión financiera provoca la evaporación del capital en pasivos ocultos (suscripciones, gastos hormiga, mora en arriendos). El cerebro humano no es apto para computar amortizaciones continuas ni mantener un balance neto actualizado de múltiples fuentes de ingresos y egresos. Antes de este sistema, los usuarios dependían de hojas de cálculo estáticas, las cuales requieren intervención manual constante, son propensas a errores humanos en las fórmulas, y no ofrecen seguridad ante el acceso no deseado (carencia de cifrado y 2FA).

La Solución Técnica

Este sistema destruye el problema de raíz mediante la automatización de flujos y la persistencia en la nube.

  • A nivel técnico, el backend intercepta las entradas a través de una API REST protegida que recalcula los agregados financieros en el servidor, devolviendo JSONs instantáneos.
  • A nivel de usuario, la interfaz renderiza los resultados sin recargas de página, empleando Vanilla JavaScript de alto rendimiento para manipular el DOM.
  • Ejemplo Real: Un usuario recibe su pago dominical. Ingresa al aplicativo por la mañana; el sistema lee su balance previo. El usuario agrega el ingreso en la modal flotante. El backend (FastAPI) ejecuta las rutinas SQL, devuelve un status 200 OK, y el frontend inmediatamente eleva la barra de estado financiero a verde, mostrando el nuevo capital.

Alcance Tecnológico y Arquitectura

Implementamos una arquitectura monolítica asíncrona con un back-end en Python (FastAPI) acoplado a un motor SQLite embebido, con capacidad de migración a bases de datos en la nube (ej., Firestore). La inyección de vistas HTML se realiza mediante Jinja2. Este stack asegura baja latencia y alta concurrencia.

SECCIÓN 2: OBJETIVOS DETALLADOS

Objetivo General

Desarrollar, desplegar y mantener un ecosistema web financiero de grado bancario que consolide la gestión integral de capitales personales (ingresos, gastos y pasivos), garantizando una disponibilidad del 99.9%, y una trazabilidad exacta de las fluctuaciones económicas del usuario mediante un diseño de interfaz altamente reactivo e intuitivo, operado bajo estrictos protocolos de ciberseguridad.

Objetivos Específicos

  1. Diseñar la Capa de Autenticación Continua: Implementar un esquema mixto de login validado por OTP (One-Time Password) inyectado por SMTP asegurando esquemas "Zero Trust".
  2. Construir un Modelo de Persistencia Atómico: Establecer un diseño relacional en SQLite que permita operaciones CRUD atómicas, asegurando que ningún fallo eléctrico corrompa las transacciones financieras.
  3. Desarrollar Dashboards Analíticos Reactivos: Programar interfaces analíticas con librerías nativas que interpreten y grafiquen la dispersión de gastos en tiempo real (Ejemplo: Gráficos de anillo y pastel).
  4. Implementar Gestión Avanzada de Deudas: Crear un algoritmo de amortización visual donde el registro de abonos a capital reduzca asíncronamente el total de cada préstamo/tarjeta.
  5. Garantizar Diseño Responsivo (Mobile-First): Utilizar directivas CSS modernas (flexbox, grid) para adaptar el portal de control a pantallas que oscilen entre los 320px hasta monitores QHD y 4K.
  6. Desplegar Arquitecturas de Alerta Predictiva: Programar lógicas y advertencias en el módulo de estudio sobre proporciones ideales (50% Necesidades, 30% Placer, 20% Ahorro).
  7. Sistematizar Suscripciones (Pagos Fijos): Otorgar un módulo tabular para la calendarización estricta de compromisos ineludibles.
  8. Habilitar Exportación Táctica de Datos: Facilitar un endpoint REST exclusivo (/api/export-expenses) de generación en memoria de Archivos SCV (Comma-Separated Values).
  9. Desarrollar Interfaz de Cotizaciones en Vivo: Implementar interacciones con APIs extranjeras y módulos News para mapear la macroeconomía en la dashboard local.
  10. Asegurar Manejo de Estados y Perfiles: Dotar al usuario de control absoluto mediante el módulo de Settings, permitiendo mutabilidad gráfica (Modo Oscuro) e ingreso a logs mediante tokens.

SECCIÓN 3: ALCANCE (ULTRA DETALLADO)

Funcionalidades Incluidas (Full Scope)

  • Módulo de Autenticación Doble (2FA): Restringe el acceso. El sistema envía tokens por socket SMPT para cruzar la frontera del Firewall interno.
  • Engine de Asimilación de Movimientos: Panel de ingesta de datos transaccionales, clasificando entre pasivos y activos, impactando directamente en la tabla movements.
  • Módulo Analítico Global (Dashboard): Pantalla centralizada para lecturas y predicciones numéricas.
  • Amortizador de Deudas: Manejo pormenorizado de obligaciones con terceros, con resta pasiva.
  • Educación 50/30/20: Calculadora pedagógica en demanda interna.

Funcionalidades NO Incluidas (Out of Scope)

  • Sincronización Bancaria Automática (Open Banking): Por restricciones de cumplimiento PSD2 y riesgo legal de intermediación en la versión 1.0, las credenciales bancarias no se cruzan.
  • Gestión Multi-Usuario: La base de datos asocia todas las rutas a un email único propietario por sesión.
  • Compra/Venta Activa de Criptomonedas: El rastreador presenta métricas (market-data), pero no emite órdenes de transacción hacia exchanges.

Limitaciones Técnicas y Supuestos

  1. Supuesto: El servidor de correo saliente (SMTP_SERVER) debe contar con una disponibilidad absoluta y no debe bloquear correos desde el dominio host (en su defecto, se usarán puentes transaccionales como SendGrid u Outlook Relay).
  2. Limitación de Vercel/Serverless: Al alojarse en arquitecturas efímeras, usar SQLite a un archivo local de disco resulta en una base de datos temporal (alocada en /tmp/). El supuesto para alta disponibilidad a largo plazo dictamina el uso de Postgre SQL o bases en nube remota para evitar amputación de datos (cold starts).

SECCIÓN 4: GLOSARIO (EXTREMO Y ACADÉMICO)

  • API (Application Programming Interface): Conjunto estandarizado de protocolos que permite a la interfaz (frontend) de nuestra plataforma "charlar" e intercambiar paquetes JSON con el servidor neuronal (FastAPI). Ejemplo: Cuando guardas un gasto en la App, tu navegador no toca la base de datos, llama a una API /api/add-movement.
  • CRUD: Acrónimo fundacional de base de datos que dictamina el ciclo de una entidad: Create, Read, Update, Delete.
  • FastAPI: Framework de Python ultrarrápido y asíncrono para el enrutamiento y despacho de recursos web, superando altamente a Flask y Django en entornos de alta escalabilidad concurrente.
  • Latencia Base: Retraso (en milisegundos) originado por distancias físicas de red o lentitud en el Query hacia la tabla SQL.
  • SMTP (Simple Mail Transfer Protocol): Transporte maestro que usa el sistema por el puerto TCP/587 (con TLS) para "disparar" correos electrónicos.
  • XSS (Cross-Site Scripting): Peligro cibernético donde un usuario malicioso intenta enviar Javascript por la caja de registro de gastos para hackear la sesión. (Mitigado en la app al sanitizar y tipar la información con plantillas Jinja2).

SECCIÓN 5: ARQUITECTURA DEL SISTEMA

Modelo Arquitectónico

El proyecto utiliza una arquitectura N-Tiers Desacoplada Lógicamente, adhiriendo al concepto del modelo Cliente-Servidor Híbrido, donde los recursos HTML iniciales son renderizados del lado del servidor (SSR - Server Side Rendering mediante Jinja2), pero toda la re-actividad de los datos en vista (DOM) ocurren a través de micro-peticiones asíncronas con promesas de Fetch de JavaScript (CSR - Client Side Rendering).

El Flujo Interno de Datos: Anatomía de un Clic

  1. Interacción UI: El usuario pulsa el botón "Guardar Gasto". Una función de Javascript asíncrona (async function) atrapa el evento (evitando el molesto recargo global).
  2. Transformación a Payload: Se toma el monto $15.00 y concepto "Cena", serializándolo a una estructura binaria de texto JSON.
  3. Petición POST (HTTP/1.1): El paquete de red abandona el navegador dirigido a la ruta remota /api/add-movement.
  4. Capa FastAPI: El protocolo Uvicorn recibe la petición, le pasa el control al Router de FastAPI que mapea la URI al método Python respectivo.
  5. Capa Inyección SQL: El archivo main.py levanta la conexión a SQLite en get_db_connection(), bloquea la tabla para escritura exclusiva.
  6. Persistencia: La tupla se inscribe en la memoria sólida / disco duro del host. Se efectúa un conn.commit().
  7. Retorno de Éxito: Se ensambla una respuesta HTTP 200 con cuerpo {"status": "success"} que viaja de nuevo hacia el navegador, informando al Javascript local que refresque los gráficos sin pestañear.

SECCIÓN 6: TECNOLOGÍAS UTILIZADAS

  • FastAPI (Backend Core): Orquestador de peticiones. Elegido por su velocidad, soporte a ASGI nativo, y tipado seguro. Validará automáticamente si intentamos enviar una letra en el monto numérico antes de llegar a la base de datos (seguridad pasiva de tipos).
  • Python 3 (Lógica de Servidor): El lenguaje universal de la ingeniería de datos. Ideal para matemáticas financieras complejas, librerías estabilizadas y sintaxis robusta.
  • SQLite (Capa de Base de Datos): Motor relacional "Zero-Configuration", empotrado directamente en disco. Ideal como puente intermedio robusto y local con alta tolerancia atómica frente a otros protocolos como Memcached.
  • Jinja2 (Motor de Plantillas): Carga los componentes y fragmentos HTML dinámicamente. Previene inyecciones peligrosas de código XSS.
  • Vanilla JS, CSS & HTML5 (Frontend): Las bases fundamentales del Internet, implementadas sin librerías pesadas como React o Angular, lo que previene tiempos masivos de compilado (Bundling), asegurando tiempos de carga (TTFB) menores a 50 milisegundos.

SECCIÓN 7: ANÁLISIS EXHAUSTIVO DE MÓDULOS

7.1 Módulo Cero: Autenticación y Despacho OTP (Login)

Propósito: Proteger la puerta principal. Ningún actor no identificado puede ejecutar comandos dentro del Dashboard o Profile. Flujo Interno:

  1. Servidor detiene a cualquier navegante en la puerta (redirect a /login).
  2. Usuario envía solicitud a /api/security/send-otp con su Mail validado.
  3. El módulo Python de aleatoriedad computa 6 números impredecibles en entropía pura.
  4. Conexión de puente a red externa mediante smtplib.SMTP(smtp_server, smtp_port). Emisión cifrada.

Trozos de Código de Ingeniería (Autenticación):

@app.post("/api/security/send-otp")
async def send_security_otp(request: Request):
    """Generación de alta entropía para credenciales temporales"""
    data = await request.json()
    email = data.get("email")
    if not email:
        return JSONResponse(content={"status": "error", "message": "Email requerido"}, status_code=400)

    # Autogenerador criptográfico pasivo
    otp = "".join([str(random.randint(0, 9)) for _ in range(6)])
    expires_at = (datetime.now() + timedelta(minutes=10)).isoformat()
    
    conn = get_db_connection()
    cursor = conn.cursor()
    cursor.execute('INSERT OR REPLACE INTO security_codes (email, code, expires_at) VALUES (?, ?, ?)', (email, otp, expires_at))
    conn.commit()
    conn.close()
    
    # Inyección real a red
    email_sent, error_msg = send_email_otp(email, otp)
  • Análisis Técnico: El código demuestra un uso estratégico del comando SQL INSERT OR REPLACE, lo que evita la proliferación caótica de códigos huérfanos anteriores, limpiando el registro antiguo por el nuevo pin, un modelo asertivo de memoria.

7.2 Módulo de Movimientos y Rutas de Retorno (Gastos e Ingresos)

Propósito: La balanza milimétrica del sistema. Absorbción de todo reporte contable humano. Flujo Interno: El Frontend intercepta los modales HTML asíncronos y emite hacia las APIs un JSON.

Trozos de Código de Ingeniería (Core Financiero):

@app.post("/api/add-movement")
async def add_movement(request: Request):
    """Motor Inserción de Vector Económico"""
    data = await request.json()
    email = data.get("email")
    movement = data.get("movement") 
    
    conn = get_db_connection()
    cursor = conn.cursor()
    cursor.execute('''
    INSERT INTO movements (user_email, type, concept, amount, date, category)
    VALUES (?, ?, ?, ?, ?, ?)
    ''', (email, movement["type"], movement["concept"], movement["amount"], movement["date"], movement["category"]))
    
    conn.commit()
    conn.close()
    return {"status": "success"}
Escenario de Recuperación Atómica: Si entre el comando `cursor.execute` y el `conn.commit()` el servidor perdiese energía eléctrica, SQLite anula todo rastro como si el código jamás hubiese sido ejecutado (Transacción). Eso previene corrupciones matemáticas "A medias".

7.3 Exportador General de Inteligencia de Datos (Reportes CSV)

Propósito: Extracción forense y exfiltración legítima de data por parte del usuario, para revisiones profundas en Excel.

@app.get("/api/export-expenses")
async def export_expenses(email: Optional[str] = None):
    # Extracción por email
    conn = get_db_connection()
    movements = conn.execute('SELECT date, concept, amount, category FROM movements WHERE user_email = ? AND type = "expense"', (email,)).fetchall()
    
    output = StringIO() # Intercambiador de Ram. No toca el disco duro (Máxima velocidad)
    writer = csv.writer(output)
    writer.writerow(["Fecha", "Concepto", "Monto", "Categoria"])
    for m in movements:
        writer.writerow([m["date"], m["concept"], m["amount"], m["category"]])
    
    output.seek(0)
    return StreamingResponse(
        iter([output.getvalue()]),
        media_type="text/csv",
        headers={"Content-Disposition": "attachment; filename=gastos.csv"}
    )
  • Análisis Técnico: Se utiliza StringIO en lugar de guardar un fichero real local evitando colapsos de disco duro, despachando el texto en memoria Ram empaquetado como cabecera (Header) de "descarga" (attachment), indicándole al navegador que expulse el documento inmediatamente (StreamingResponse) sin corromper la latencia frontal.

SECCIÓN 8: DIAGRAMACIÓN Y BASE DE DATOS

La persistencia del esquema recae sobre seis bóvedas SQL ultrarrápidas y conectadas por referenciabilidad en caso de escalar (Primary Keys vs Foreign Keys).

  1. movements: id (PK, Numérico Irrepetible), user_email (Text, Vínculo de usuario), type (Ingreso/Gasto), concept, amount, date, category.
  2. user_profiles: user_email (PK, Centro de verdad del usuario), first_name, phone_number.
  3. debts (Acreencias Externas): id, user_email, name, total_amount, paid_amount (Almacena el diferencial aportado), due_date.
  4. security_codes: email (PK, unifica un único código en vuelo por receptor), code, expires_at.

SECCIÓN 9: LÓGICA PASO A PASO DEL SISTEMA (EXPLICACIÓN PEDAGÓGICA Y ALGORÍTMICA)

¿Cómo Calcula el Sistema el Balance de Saldo Total?

El Frontend es el ejecutor principal del consolidado. Llama a la ruta global /api/movements obteniendo un mega-bloque JSON (Arreglo Base).

  1. Crea una variable local (Acumulador) let currentBalance = 0; y let totalIncome = 0;.
  2. Genera un Ciclo For masivo en memoria. movements.forEach(transaccion => { ... })
  3. Si el campo lógico es "income", le fuma valor positivo al acumulador.
  4. Si el campo lógico es "expense", lo sustrae brutal y enteramente.
  5. Emite la cifra renderizada y animada sobre el nodo document.getElementById('displayTotalBalance').innerText = currentBalance;.

Escenarios de Estrés / Error Inminente

  • ¿Qué pasa si el Internet del usuario se pierde al mandar un pago de deuda? El Fetch local dispara un evento lógico .catch(error => alert('Network Disconnected...')) cancelando la operación visual sin engañar al usuario, evitando contabilidad fantasma.

SECCIÓN 12 Y 13: CIBERSEGURIDAD, PROTECCIÓN DE DATOS Y RESPALDO

Postura de Ataque (Red Team / Pentesting Previsto)

  • Inyección SQL Blindada: Debido a que utilizamos motores de comandos paramétricos de tipo (?, ?, ?) y una tupla (email, val1, val2), es literalmente imposible a nivel lógico que un ciberdelincuente agregue comandos fatales (como un infame DROP TABLE) dentro del campo email.
  • Ataques XSS / Cross Protocol: Las respuestas en HTML son procesadas, el DOM escapa etiquetas (<script>).
  • Protección Persistente / Backups (Respaldo Forense): La arquitectura es agnóstica a sistemas fijos y guarda todo centro de información concentrado orgánicamente en finanzas_personales.db. Se indica a los técnicos de despliegue programar herramientas de duplicado rsync, CronJobs unificados que copien asíncronamente el archivo hacia un almacén de cinta o clúster S3 AWS sin interrumpir las tablas operativas.

SECCIÓN 14: AUDITORÍA DE CONSOLA Y SEGUIMIENTO LOCAL (FORENSICS LOGGING)

El archivo main.py incorpora interceptadores explícitos que mapean en memoria los fallos que las excepciones globales (@app.exception_handler(Exception)) intentan evadir, logrando que cada caída sea pintada en Terminal por el servidor (CRITICAL VERCEL ERROR: ...). Al combinarse con lecturas y trazos asíncronos que muestran códigos (ESTADO DE ENVIO EMAIL: EXITOSO / FALLIDO) en ambiente terminal o Vercel-Logs Dashboard, los encargados del soporte del código aseguran reparaciones sub-hora.

[!TIP] COMPATIBILIDAD VERCEL (SERVERLESS DEPLOYMENT) El bloque lógico detecta la variable VERCEL="1". Si el código es succionado hacia la nube Vercel, muta automáticamente toda la red de rutas, ignorando arranques locales en hilos, ruteando la DB hacia la partición exclusiva /tmp del servidor, demostrando un comportamiento inteligente de multi-sistema inigualable y adaptabilidad para equipos SysAdmins.
